Singra Vault — Phasensteuerung für KI-Agenten
Zweck dieses Dokuments
Dieses Dokument ist kein Ersatz für das detaillierte Architekturkonzept „Singra Vault — Neues
Integrations-, Quarantäne- und Manipulationsschutzsystem“.
Das Architekturkonzept bleibt die fachliche und sicherheitstechnische Grundlage. Dieses Dokument
dient nur dazu, die Umsetzung in klar getrennte, kontrollierbare Phasen aufzuteilen, damit ein KI-
Modell immer nur einen begrenzten Auftrag erhält und nicht versehentlich das gesamte System
gleichzeitig umbaut.
Die Grundregel lautet:
Ein KI-Agent arbeitet immer nur an genau einer Phase. Alles außerhalb dieser Phase ist
verboten, außer es wird ausdrücklich als Vorprüfung, Test oder Dokumentation verlangt.

Phase 3 — Repository-Layer und Feature-Flag,
ohne UI
Ziel
Phase 3 baut die technische Client-Schnittstelle zu den neuen Supabase-RPCs und Tabellen. Sie
verbindet die neuen reinen Module aus Phase 1 mit den RPCs aus Phase 2, aber noch nicht mit der
normalen UI, Autofill, Export oder dem alten VaultContext.
Phase 3 ist die erste Phase mit echter Client-Integration, aber sie bleibt hinter einem Feature-Flag und
darf keine bestehenden Produktivpfade ersetzen.
Hauptziel in einem Satz
Baue einen isolierten vaultOpLog -Repository-/Sync-Zugriff, der RPCs typisiert aufruft,
Operationen sicher serialisiert/deserialisiert und DB-Roundtrips gegen Canonicalization
testet — ohne UI- oder Runtime-Umbau.

Phase 9 — UI-, Quarantäne-, Safe-Mode- und
Konflikt-Anbindung hinter Feature-Flag
Ziel
Phase 9 bindet die neue State Machine an sichtbare UI-Zustände an, aber zunächst nur hinter Feature-
Flag.
Hauptziel in einem Satz
Zeige verified, conflict, quarantine, safeMode und lockedCritical korrekt in der UI an,
ohne Autofill/Export/Suche unsicher freizugeben.
Erlaubte Arbeiten
Der Agent darf:
neue Vault Security Modes anzeigen.
Quarantäne-Bereich an neue Gründe anbinden.
Konflikt-UI anbinden.
Safe-Mode-Anzeige anbinden.
Kategorie-Quarantäne anzeigen.
Restore/Delete/Resolve-Aktionen als signierte Operationen auslösen.
Tests für UI-Zustände schreiben.
Verbotene Arbeiten
Der Agent darf nicht:
Feature-Flag entfernen.
Quarantäne-Records entschlüsseln.
generische Accept-Aktion implementieren.
Autofill für containerQuarantined freigeben, wenn erste Version konservativ bleiben soll.
alte Logik bereits löschen, wenn Migration/Shadow Mode nicht stabil sind.
Sicherheitsregeln
Normale Liste zeigt nur verified Records.
Quarantäne zeigt keine unsicheren Plaintexts.
Konflikte sind nicht Quarantäne.
Kategoriefehler blockieren nicht den gesamten Vault.
lockedCritical nur bei Root-Problemen.
Jede UI-Aktion erzeugt konkrete signierte Operation.

Phase 11 — Alte Runtime-Logik entfernen
Ziel
Phase 11 entfernt alte Sicherheitslogik aus Produktivpfaden, nachdem das neue System getestet,
migriert und hinter Feature-Flag stabil ist.
Hauptziel in einem Satz
Entferne alte Rebaseline-, Snapshot-Digest-, TTL- und Kategorie-Globalblockade-Logik aus
Runtime-Pfaden, ohne Migration-Recovery zu zerstören.
Erlaubte Arbeiten
Der Agent darf:
alte automatische Rebaseline entfernen.
alte Snapshot-Digest-Vertrauensentscheidung entfernen.
LOCAL_WRITE_CACHE_TTL_MS  entfernen.
Kategorie-Globalblockade entfernen.
direkte alte Vault-Writes entfernen.
alte Tests umschreiben oder löschen, wenn sie altes Verhalten erwarten.
Legacy-Code auf Migration-Service begrenzen.
Verbotene Arbeiten
Der Agent darf nicht:
Migration-Service zerstören.
Legacy-Daten unlesbar machen, bevor Migration abgeschlossen ist.
neue Sicherheitsgates entfernen.
Feature-Flag zu früh entfernen, falls Rollout noch nicht abgeschlossen ist.
Sicherheitsregeln
Keine alte Runtime-Sicherheitsentscheidung darf parallel weiterleben.
Legacy-Code nur Migration-Service.
Keine auskommentierten Altpfade.
Keine Compatibility-Bypässe.
Tests dürfen altes Verhalten nicht mehr als korrekt bestätigen.

3. Standard-Prompt-Vorlage für jede Phase
Diese Vorlage kannst du verwenden, wenn du einem KI-Modell eine Phase gibst.
Du arbeitest im Repository von Singra Vault auf dem Branch feature/vault-
operation-log-quarantine-integrity.
Arbeite ausschließlich an PHASE X: <Phasenname>.
Das Architekturkonzept bleibt verbindlich. Ändere das Sicherheitskonzept 
nicht, außer du dokumentierst einen klaren Widerspruch und stoppst zur 
Rückfrage.
Verboten:
- keine Arbeiten außerhalb dieser Phase,
- keine UI-Anbindung, sofern diese Phase es nicht ausdrücklich erlaubt,
- keine Autofill-/Export-/Such-/Clipboard-Änderungen, sofern diese Phase es 
nicht ausdrücklich erlaubt,
- keine Migration, sofern diese Phase es nicht ausdrücklich erlaubt,
- keine Entfernung alter Runtime-Logik, sofern diese Phase es nicht 
ausdrücklich erlaubt,
- keine direkten Vault-Tabellenwrites außerhalb der Operation-Schicht,
- keine Secrets loggen,
- keine Tests löschen, um grün zu werden.
Vor Beginn:
1. Prüfe Branch und Git-Status.
2. Liste die Dateien auf, die du voraussichtlich änderst.
3. Liste die Sicherheitsinvarianten auf, die diese Phase betrifft.
4. Stoppe, wenn uncommitted Änderungen existieren, die nicht zu dieser Phase 
gehören.
Während der Umsetzung:
- kleine, nachvollziehbare Commits,
- Tests direkt mitimplementieren,
- keine stillen Bypässe,
- keine auskommentierten Altpfade,
- keine neue Dependency ohne Begründung.
Am Ende:
1. Nenne geänderte Dateien.
2. Nenne neue oder geänderte Services.
3. Nenne entfernte oder bewusst nicht entfernte Altpfade.
4. Nenne Tests mit Ergebnis.
5. Nenne bekannte pre-existing Failures getrennt.
6. Nenne nicht verifizierte Annahmen.
7. Nenne Risiken und nächste Phase.

6. Stop-Regeln für KI-Agenten
Ein KI-Agent muss stoppen und Rückfrage stellen, wenn:
er außerhalb der aktuellen Phase arbeiten müsste,
er alte Runtime-Logik entfernen müsste, obwohl die Phase das nicht erlaubt,
er UI/Autofill/Export/Suche/Clipboard ändern müsste, obwohl die Phase das nicht erlaubt,
er eine neue Dependency für Kryptografie vorschlägt,
er Secrets oder echte Nutzerdaten sieht,
er Migrationen ausführen müsste, obwohl die Phase das nicht erlaubt,
er Tests löschen müsste, um grün zu werden,
er das Sicherheitskonzept ändern müsste,
er nicht erklären kann, welche signierte Operation eine Nutzeraktion erzeugt,
er nicht beweisen kann, dass manipulierte Records nicht entschlüsselt werden.
